# 创意喵单一视频平台收敛设计

## 背景

当前仓库存在明显定位冲突：

- 对外文案已经把创意喵定义成“视频创意任务平台”。
- 产品实现仍按“多分类创意任务市场”运行，商家可以发布文案、设计、摄影、音乐、开发等任务，创作者可以按分类筛选。

这种状态会导致三类问题：

1. 用户认知错位：落地页和说明承诺的是视频平台，但真正操作时看到的是泛创意众包市场。
2. 产品决策失真：业务口径说“创意=视频”，代码和测试却继续维护多分类能力。
3. 后续维护反复回弹：只改文案不改逻辑，未来任何页面和测试都可能再次把多分类暴露出来。

## 目标

将创意喵彻底收敛为单一“视频任务平台”，让文案、文档、页面行为、后端默认规则、测试口径保持一致。

## 非目标

- 不在本次做数据库迁移或历史任务批量清洗。
- 不在本次引入新的复杂视频专属字段模型，例如脚本文件、样片上传、时长约束结构化字段。
- 不额外重构无关业务模块。

## 决策

采用“产品单一化，数据兼容保留”的策略：

- 对用户可见层，移除“多分类创意平台”的表达和交互。
- 对后端兼容层，短期保留 `category` 字段，但其业务意义固定为“视频”。
- 对旧调用兼容，服务端在创建任务时强制写入视频分类值，不再信任前端传入的其他分类。

## 设计范围

### 1. 文案与产品定位

- 统一使用“视频任务平台”或“视频创意任务平台”。
- 去掉“文案创作 / 设计创意 / 摄影拍照 / 音乐音频 / 程序开发”等平台能力暗示。
- 所有帮助性描述改为围绕视频创作交付，例如：
  - 视频脚本与口播要求
  - 拍摄要求与镜头说明
  - 剪辑节奏与成片规范
  - 交付格式与验收标准

### 2. 商家发单流程

- 删除“任务分类”选择控件。
- 页面直接声明该任务为视频任务。
- 发布请求仍可发送 `category`，但前端固定为视频值，后端也会兜底强制为视频。
- 描述输入提示改为视频任务专用文案。

### 3. 创作者任务大厅

- 删除按分类筛选。
- 保留价格、排序、关键词搜索。
- 统计、列表、空状态、筛选标签等不再出现分类相关信息。
- 任务列表与详情页不再展示分类 badge。

### 4. 公共任务大厅

- 公开浏览页同步删除任务分类筛选和分类标签。
- 对外说明统一成“浏览可认领的视频任务”。

### 5. 后端与模型

- `TaskCategory` 保留以避免大范围破坏，但明确标记当前平台仅支持视频任务。
- 创建任务时忽略非视频分类输入，统一按视频分类保存。
- 查询接口继续兼容 `category` 参数，但产品前端不再使用；如有必要可在实现中忽略该参数，确保不会形成对外多分类能力。

### 6. 测试调整

- 删除或改写基于多分类的筛选测试。
- 重点覆盖：
  - 任务创建默认落为视频分类
  - 任务大厅在无分类筛选下仍可正常加载
  - 关键词 / 排序 / 价格筛选不受影响

## 受影响文件

- 文档：`README.md`、`docs/V1_0_PRD.md`
- 模型 / 后端：`internal/model/task.go`、`internal/handler/business.go`、`internal/handler/creator.go`
- 页面模板：
  - `web/templates/business/task_create.html`
  - `web/templates/creator/task_hall.html`
  - `web/templates/tasks.html`
  - `web/templates/creator/task_detail.html`
  - `web/templates/business/task_list.html`
- 测试：`test/test_task_filters.sh`

## 风险与处理

### 风险 1：历史非视频分类任务仍存在

处理：
- 本次先不做数据迁移。
- UI 不再展示分类分支，统一按视频任务口径展示。
- 若后续需要清洗历史数据，再单独做迁移任务。

### 风险 2：旧脚本或旧前端继续传非视频分类

处理：
- 后端创建任务时强制写死为视频分类，避免继续生成脏数据。

### 风险 3：已有未提交改动冲突

处理：
- 修改时仅触碰本次目标相关文件。
- 变更前先读取现有文件内容，避免覆盖工作区里的用户改动。

## 验收标准

- 仓库主文档不再把平台描述成多分类创意市场。
- 商家发单页没有任务分类选择，且文案围绕视频任务。
- 创作者任务大厅与公开任务大厅不再暴露分类筛选。
- 任务列表与详情页不再展示多分类标签。
- 创建任务在后端统一落为视频分类。
- 相关测试与脚本不再依赖多分类行为。
