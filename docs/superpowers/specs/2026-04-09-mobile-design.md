# 移动端独立产品线设计文档

## 背景
当前创意喵已有桌面端/后台式页面结构，角色区分较重，页面信息组织更偏平台管理视角。现在需要新增一个独立的移动端产品线，以 `/mobile/` 为统一前缀，面向手机访问场景，整体体验改为更偏 C 端 app 风格。

本次设计目标是：在不重写后端核心能力的前提下，基于现有 API 和业务能力，落地一套独立的移动端前端信息架构与页面体系。

## 目标
- 新增独立的 `/mobile/` 前端入口，不直接覆盖现有桌面端页面。
- 弱化商家/创作者角色感知，不提供角色切换入口。
- 采用底部 Tab 的移动端主导航结构。
- 首页固定为任务大厅。
- 第二个 Tab 为过审作品展示。
- 第三个 Tab 为“我的”，聚合发布任务、领取任务、钱包等个人能力入口。
- 整体视觉改造为更偏 C 端产品、接近 app 的界面风格。

## 非目标
- 不重写现有后端权限模型与核心业务模型。
- 不在本阶段统一重构桌面端页面。
- 不在本阶段引入新的复杂交易流或新的角色系统。
- 不在本阶段做完全独立于现有模板体系之外的前端技术栈迁移。

## 设计原则
1. **独立产品感**：移动端不是桌面端缩窄版，而是单独的信息架构与视觉体系。
2. **弱角色化**：用户进入移动端后，不再被要求理解“当前角色”，而是围绕任务、作品、个人中心来使用产品。
3. **内容化优先**：首页任务大厅采用推荐流思路，而不是传统后台式列表优先。
4. **复用现有能力**：优先复用现有 API、登录状态和业务数据结构，把改造重点放在模板层和前端交互层。
5. **移动端优先**：全站围绕手机视口设计，使用底部导航、卡片流、吸底操作、轻筛选等移动端模式。

## 信息架构
移动端主导航固定为 3 个一级 Tab：

### 1. 任务大厅
作为首页与默认落点。

职责：
- 展示推荐任务流
- 承载轻量搜索和轻筛选
- 提供任务详情与领取/参与入口

特征：
- 首页形态更接近内容推荐流
- 任务卡使用更强视觉层级
- 保留必要筛选，但筛选为辅助，不喧宾夺主

### 2. 过审作品
作为独立内容展示页。

职责：
- 展示已过审作品/优秀案例
- 强化创作者成果展示与平台内容氛围
- 支持从作品回看关联任务

特征：
- 视觉上更接近内容社区或灵感流
- 支持卡片流/双列流等更具展示感的布局
- 强调图片、标题、结果状态、任务来源

### 3. 我的
作为个人中心与个人能力聚合页。

职责：
- 展示用户基础信息与状态
- 聚合“发布任务”“领取任务”“钱包”等功能入口
- 汇总个人任务、个人作品、钱包记录等数据

特征：
- 用宫格入口 + 列表入口组织功能
- 避免再次暴露角色切换
- 将原本分散在商家端/创作者端的个人功能收束为账户中心能力

## 页面设计

### 任务大厅页
定位：推荐流首页。

建议结构：
1. 顶部品牌区 / 问候区
2. 搜索框
3. 轻量分类筛选（chips/tag）
4. 推荐任务卡片流
5. 底部 Tab

任务卡片设计建议：
- 大圆角卡片
- 可包含封面图、渐变背景或品类插画占位
- 核心信息包括：任务标题、预算、截止时间、标签、任务状态
- 主操作按钮明确，如“查看详情”“立即领取”
- 单卡信息密度控制在移动端可快速浏览范围内

交互原则：
- 推荐流为主，筛选为辅
- 不使用桌面端左侧边栏或双栏结构
- 不做复杂后台式筛选面板，筛选放在顶部轻量区域

### 过审作品页
定位：案例与内容展示页。

建议结构：
1. 顶部标题与说明
2. 筛选或分类（可选，轻量）
3. 作品流展示区
4. 底部 Tab

作品卡片建议：
- 支持更大的视觉展示面积
- 可采用双列流或瀑布流风格
- 展示作品封面、作品标题、所属任务、奖励/过审标识等

交互原则：
- 更偏“看内容”和“看案例”
- 允许从作品跳转至关联任务或任务详情
- 视觉气质应比任务大厅更偏内容社区

### 我的页
定位：账户与操作聚合页。

建议结构：
1. 用户信息卡（头像、昵称、核心数据）
2. 常用功能宫格入口
3. 个人数据模块
4. 钱包/记录/设置等列表入口
5. 底部 Tab

常用功能入口至少包含：
- 发布任务
- 领取任务
- 钱包

可扩展入口：
- 我的任务
- 我的作品
- 收益明细
- 设置

交互原则：
- 高优先功能上移，适合单手点击
- 信息与入口混排，但层级清晰
- 视觉上更像 C 端账户页，而不是管理后台目录

## 视觉风格
已确认方向：**小红书感、偏内容化、偏 app 风格**。

具体风格要求：
- 使用更柔和、明亮的底色体系
- 卡片为主，圆角更大，阴影更轻
- 更强调图片感、封面感、内容感
- 页面留白更充足，信息分组更像消费级产品
- CTA 按钮、Tab、搜索框等都要更贴近原生 app 视觉语言

明确避免：
- 桌面端后台感
- 左侧导航栏结构
- 过密表格与过重边框
- 强角色切换提示

## 组件与交互规范

### 主框架组件
- 移动端顶部栏
- 底部 Tab Bar
- 卡片流容器
- 轻量搜索框
- 分类 Chips
- 宫格入口模块
- 列表入口模块

### 交互要求
- 底部 Tab 固定为全局导航
- 页面主操作按钮清晰、可单手点击
- 任务卡/作品卡可整卡点击
- 统一使用移动端过渡与反馈，不延续桌面端交互习惯
- 重要操作优先使用卡片内按钮、吸底按钮或明显 CTA

## 技术落点

### 路由与目录建议
新增独立的移动端模板与资源组织，建议以 `/mobile/` 为入口前缀。

首批页面建议包括：
- `/mobile/index.html`：任务大厅
- `/mobile/showcase.html`：过审作品
- `/mobile/me.html`：我的

从“我的”继续进入二级页：
- `/mobile/task/publish.html`
- `/mobile/task/claimed.html`
- `/mobile/wallet.html`
- `/mobile/transactions.html`

目录建议：
- `web/templates/mobile/`：移动端模板
- `web/static/mobile/`：移动端专用样式与脚本

### 数据与接口策略
- 登录态沿用现有认证方式
- 优先复用现有 API 与返回结构
- 若桌面端接口字段不完全适配移动端展示，优先做前端适配；仅在必要时补充轻量接口字段
- 不因移动端改版而引入新的业务状态机

## 实施范围建议
首批实现范围：
1. 搭建 `/mobile/` 独立导航框架
2. 实现 3 个一级 Tab 页面
3. 将任务大厅改造成推荐流卡片首页
4. 将过审作品做成内容展示流
5. 将“我的”做成聚合个人能力的账户页
6. 接通发布任务、领取任务、钱包等二级入口

## 风险与约束
1. 现有部分页面和接口命名带有明显角色语义，移动端前端层需要做重新组织与文案弱化。
2. 如果现有数据接口缺少封面图、摘要、展示标签等内容化字段，移动端卡片视觉表现可能需要先用占位策略处理。
3. 如果继续沿用桌面端模板片段，容易回退成“PC 页面缩窄版”，实现时需严格隔离移动端样式与布局。
4. 移动端与桌面端并存时，模板、脚本和样式应明确分层，避免互相污染。

## 推荐实施策略
推荐采用“独立移动端前端壳 + 复用现有业务 API”的策略：
- 前端层新建 `/mobile/` 页面体系
- 桌面端页面保持现状
- 移动端重新组织信息架构与视觉样式
- 后端只做必要的最小适配

这是当前最稳妥且最符合目标的方案，既能做出独立产品感，又能控制改造范围。
