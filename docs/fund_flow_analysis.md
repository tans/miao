# 资金相关流程与接口分析（现状 + 优化建议）

## 1. 资金主流程梳理

### 1.1 充值流程（微信支付）

1) 用户调用 `POST /api/v1/account/recharge` 创建充值订单。服务端落库 `payment_orders`，状态为 `pending`。  
2) 若开启微信支付，调用微信统一下单，返回 `code_url`。  
3) 微信回调 `POST /api/v1/payment/callback`，系统将订单置为 `paid`，并调用 `creditUserBalance` 给用户加余额并记交易流水（type=1）。  
4) 用户可通过 `GET /api/v1/account/recharge/:order_no` 查询订单状态，若仍待支付会主动查微信单号并尝试更新。

对应实现：`internal/handler/payment.go`、`internal/repository/payment.go`、`internal/service/wechat_pay.go`。

### 1.2 商家发任务预冻结

1) 商家创建任务 `POST /api/v1/business/tasks`。  
2) 系统计算总预算：`total_budget = total_count * (unit_price + award_price)`。  
3) 通过 `CreateTaskWithFreeze` 在一个事务里：创建任务、扣减可用余额、增加冻结金额、写冻结交易（type=3）、增加发布次数。

对应实现：`internal/handler/business.go`、`internal/repository/business.go`。

### 1.3 作品验收结算（通过/拒绝/举报）

商家在 `PUT /api/v1/business/claim/:id/review` 对认领单结算：

- 通过：创作者获得「参与奖励净额 + 采纳奖励」，平台获得参与奖励抽成，商家冻结资金相应解冻并转为已支付。  
- 拒绝：创作者仅得参与奖励净额，平台拿参与奖励抽成，采纳奖励不支付。  
- 举报：创作者不获奖励，参与奖励归平台（罚款），采纳奖励返还商家。

对应实现：`internal/handler/business.go`、`internal/repository/business.go`。

### 1.4 任务取消退回

商家取消任务 `DELETE /api/v1/business/tasks/:id`：  
1) 计算可退金额（当前实现基于 `total_budget - paid_amount`）；  
2) 取消 pending 认领并处理保证金返还；  
3) 商家余额回补，冻结金额减少，写解冻流水（type=4）；  
4) 任务状态置为 cancelled。

对应实现：`internal/handler/business.go`。

### 1.5 提现流程

创作者提现 `POST /api/v1/creator/withdraw`：  
1) 事务内锁用户并校验可提现余额；  
2) 扣减余额并写提现交易（type=6）；  
3) 返回到账金额与手续费信息。

对应实现：`internal/handler/account.go`。

## 2. 主要资金接口清单（按业务域）

### 2.1 C端/商家可见
- `POST /api/v1/account/recharge` 创建充值订单（微信下单）。
- `GET /api/v1/account/recharge/:order_no` 查询充值订单状态。
- `POST /api/v1/payment/callback` 支付回调。
- `POST /api/v1/creator/withdraw` 创作者提现。
- `GET /api/v1/creator/transactions` 创作者交易流水。
- `GET /api/v1/business/transactions` 商家交易流水。
- `POST /api/v1/business/tasks` 发任务并冻结预算。
- `PUT /api/v1/business/claim/:id/review` 审核并结算。
- `DELETE /api/v1/business/tasks/:id` 取消任务并退回。

### 2.2 管理端财务
- `GET /api/v1/admin/finance/stats` 财务统计。
- `GET /api/v1/admin/finance/transactions` 财务交易明细。
- `GET /api/v1/admin/finance/transactions/:id` 交易详情。

## 3. 当前实现中的关键风险点

### 3.1 回调验签与幂等性不足（高优先级）
- 支付回调仅按 body 解析并处理，未看到对微信回调签名/证书的校验。  
- 订单状态更新和余额入账是分离步骤，尽管有 `pending -> paid` 条件更新，但“订单更新成功 + 入账失败”会造成对账不一致。

### 3.2 金额精度使用 `float64`（高优先级）
- 资金字段与计算大量使用 `float64`，存在精度漂移风险（充值、提现、结算、冻结/解冻）。

### 3.3 多步资金变更缺少统一事务（高优先级）
- `ReviewClaim` 内部包含多次余额/冻结/任务金额/流水写入，现状多为分散调用，非单事务，部分调用未检查错误返回，存在部分成功风险。

### 3.4 提现佣金计算逻辑疑似单位错误（高优先级）
- `GetCommission()` 返回如 `0.10`（10%），提现中却按 `commissionRate/100` 再次缩放，导致手续费近似 0.1% 而不是 10%。

### 3.5 微信支付签名实现存在可用性风险（高优先级）
- 签名调用 `rsa.SignPKCS1v15(nil, nil, ...)` 传入空私钥，运行时会失败；即使在模拟场景可绕开，真实支付链路不可用。

### 3.6 接口语义与历史兼容路径混用（中优先级）
- 同一路径 `/account/recharge` 同时存在“模拟充值”和“支付下单”两套处理语义，易造成客户端与文档理解偏差。

### 3.7 审计事件模型不够完善（中优先级）
- 目前交易流水可覆盖多数场景，但缺少“资金事件唯一键/业务幂等键/来源渠道”字段，跨系统对账成本高。

## 4. 优化建议（按落地优先级）

## P0（立即改，先保资金安全）

1) **统一“资金变更事务边界”**  
   将充值入账、提现、审核结算、任务取消退款改造为“单服务函数 + 单数据库事务”。任何一步失败全部回滚。

2) **引入幂等键与防重机制**  
   - 支付回调：`out_trade_no + transaction_id + event_id` 建唯一索引；  
   - 业务结算：`claim_id + action(approve/reject/report)` 建幂等锁。

3) **金额改为整数分（cent）**  
   数据库新增 `*_cent BIGINT`，代码统一整型运算，展示层再格式化为元。

4) **修复提现佣金计算**  
   将提现到账计算统一为：`actual = amount * (1 - commissionRate)`（commissionRate 已是 0.10 形式）。

5) **修复微信签名与回调验签**  
   使用正确私钥签名；校验微信回调签名和时间戳，防重放。

## P1（1~2 个迭代，提升稳定性）

1) **建立账务“双分录思维”模型（最小化版）**  
   至少确保每笔资金事件都有“来源账户 + 去向账户 + 事件类型 + 业务ID + 幂等ID”。

2) **补齐失败补偿/重试机制**  
   支付入账失败时落“待补偿表”，由定时任务自动补偿；所有补偿必须幂等。

3) **增强对账能力**  
   新增日对账任务：微信订单、`payment_orders`、`transactions` 三方核对并输出差异报告。

4) **统一接口语义**  
   - 保留一个充值入口（推荐 `POST /account/recharge/orders`）；  
   - 模拟充值仅在测试环境开放，且路径显式如 `/internal/mock/recharge`。

## P2（中长期，提升扩展性）

1) **提炼 `WalletService` / `LedgerService`**  
   将“更新余额 + 冻结 + 流水 + 业务状态”沉淀为领域服务，handler 只做参数与鉴权。

2) **可观测性建设**  
   - 每笔资金操作打 `trace_id`、`biz_id`；  
   - 指标：充值成功率、回调重试数、补偿成功率、对账差异数。

3) **风控规则层**  
   提现限频、单日额度、异常任务结算预警等。

## 5. 推荐改造顺序（四周版）

- **第1周**：提现佣金修复 + 微信签名修复 + 回调验签 + 幂等索引。  
- **第2周**：审核结算/任务取消改单事务；补齐错误处理。

## 7. 关于 P0「事务边界」与“提现到账后回滚”的说明

你提到的核心问题非常关键：

> 如果“提现已经到账”，但系统里其他步骤失败后回滚，是否会产生超发？

答案是：**会**，如果把“外部打款成功”和“本地数据库回滚”混在一个“看似原子”的流程里处理，就会产生资金不一致风险。  
所以 P0 的“事务边界”要明确分两层：

### 7.1 先定义边界：数据库事务只覆盖“本地状态原子性”

数据库事务（`BEGIN ... COMMIT/ROLLBACK`）只能保证：
- 用户余额、冻结金额、交易流水、提现单状态等**本地表**一致；
- 失败时本地一起回滚，不留“半成功”的本地脏数据。

它**不能**回滚第三方侧的动作（如微信/银行已打款）。

### 7.2 正确拆分提现流程（避免超发）

推荐把提现拆成三段状态机，而不是“一次接口做完所有动作”：

1) **申请阶段（本地事务）**  
   - 校验可提现余额；
   - 先扣减可用余额（或转入“提现中冻结”）；
   - 生成 `withdraw_order`（状态 `processing`）+ 交易流水；
   - 事务提交后才返回“申请成功”。

2) **出款阶段（调用第三方）**  
   - 异步 worker 按 `withdraw_order` 调用三方打款；
   - 调用必须带业务幂等号（`withdraw_no`），确保重试不重复出款。

3) **回执阶段（本地事务）**  
   - 成功回执：`processing -> success`（仅更新状态，不二次扣款）；
   - 失败回执：`processing -> failed`，并做“余额返还/解冻”补偿流水；
   - 全程使用状态条件更新（CAS）防止并发重复处理。

这样即使“第三方已成功打款”，本地也不会通过回滚把扣款撤掉导致超发；
本地只会把订单状态推进到 `success`，保持“钱已经出去了”的账实一致。

### 7.3 如何保证“回滚正确性”

要点是“**不把不可回滚的外部动作放进可回滚的本地事务语义里**”。可执行做法：

1) **状态机 + 条件更新**  
   - 例如：`UPDATE withdraw_orders SET status='success' WHERE id=? AND status='processing'`；
   - 受影响行数必须为 1 才算成功，避免重复入账/重复回退。

2) **幂等键强约束**  
   - `withdraw_no`、`channel_txn_id` 建唯一索引；
   - 回调/轮询重试只会命中同一笔订单。

3) **Outbox / 补偿任务**  
   - 本地事务里写 `outbox_event`；
   - 事务提交后再投递“调用三方”任务；
   - 投递失败可重试，不影响本地账务原子性。

4) **“先记账后出款”或“先冻结后出款”一致策略**  
   - 不允许“先出款后记账”；
   - 账务先落地，出款失败再通过补偿回退。

5) **日终对账兜底**  
   - 三方出款单 vs 本地 `withdraw_orders` vs `transactions` 每日核对；
   - 出现差异自动生成工单与补偿任务。

### 7.4 一句话总结

P0 的“事务边界”不是“把所有动作包进一个事务”，而是：

- **本地可控数据**：强事务原子；  
- **外部不可控动作**：幂等调用 + 状态机推进 + 补偿对账。  

这套方式可以避免你担心的“提现已到账但本地回滚导致超发”。

## 8. 「简单优先」的优化方案（先小改，后重构）

如果当前团队人力有限，建议采用 **Minimal Change（最小改动）** 路线，优先做“高收益、低侵入”的修复，再逐步演进到完整账务体系。

### 8.1 目标与原则

- **不大改表结构、不重写领域层**，先把资金正确性风险降到可控；
- 每次上线只解决一类问题，确保可灰度、可回滚、可观测；
- 优先保证“不会多发、不会少扣、不会重复结算”。

### 8.2 第一阶段（1~3 天）：快速止血

1) **修提现佣金计算单位**
   - 把提现公式统一为：`actual = amount * (1 - commissionRate)`（commissionRate=0.10 表示 10%）。
   - 增加 3 组单测：10%、5%、3% 档位。

2) **修微信签名调用**
   - 使用真实私钥对象签名，不允许 `nil` key；
   - 对签名失败直接报错并打日志，避免“静默失败”。

3) **关键资金操作补错误处理**
   - `ReviewClaim`、`CancelTask` 中所有 `Update*/CreateTransaction` 必须检查 error；
   - 任一步失败立即返回，避免继续执行造成部分成功。

> 这一步不改变业务流程，只补明显 bug 和错误兜底，风险最低。

### 8.3 第二阶段（3~5 天）：最小幂等与防重

1) **支付回调防重（不改大流程）**
   - `payment_orders` 增加唯一约束（如 `order_no` 已有则确认）；
   - 回调处理先做状态条件更新：仅 `pending -> paid` 时继续入账。

2) **提现防重复处理**
   - 新增 `withdraw_orders`（最简字段：`withdraw_no`、`user_id`、`amount`、`status`、`channel_txn_id`）；
   - `withdraw_no`、`channel_txn_id` 唯一索引；
   - 回调/重试通过 `status=processing` 条件推进到 `success/failed`。

3) **关键接口加请求幂等键（可选但推荐）**
   - 在网关或 handler 支持 `Idempotency-Key`；
   - 重复请求直接返回首次结果。

### 8.4 第三阶段（1 周）：小范围事务收口

1) **先收口两个高风险接口**
   - `PUT /business/claim/:id/review`
   - `DELETE /business/tasks/:id`

2) **做法**
   - 在 service 层包一层事务函数；
   - 把“余额、冻结、任务金额、流水、状态”放进同一 DB 事务；
   - 不在事务里调用第三方（第三方动作异步化）。

> 只收口两个最复杂接口，避免一次性重构全部资金链路。

### 8.5 第四阶段（后续迭代）：再做结构化升级

- 金额从 `float64` 迁移到 `cent` 整数；
- 引入 outbox、日对账、补偿任务；
- 再考虑 `WalletService/LedgerService` 的体系化重构。

### 8.6 简化版里程碑（建议）

- **M1（本周）**：修公式 + 修签名 + 补错误处理。  
- **M2（下周）**：支付/提现最小幂等、防重落地。  
- **M3（第3周）**：`ReviewClaim` 与 `CancelTask` 事务化收口。  
- **M4（第4周+）**：cent 化 + 对账/补偿 + 领域重构。

### 8.7 “简单优先”验收标准

- [ ] 同一支付回调重复 100 次，只入账 1 次；  
- [ ] 同一提现回调重复 100 次，只成功 1 次；  
- [ ] `ReviewClaim` 任一步失败，不出现余额与流水不一致；  
- [ ] 线上错误日志可定位到 `order_no/withdraw_no/claim_id`。
